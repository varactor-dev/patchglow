# CLAUDE.md — Project Instructions for Claude Code  ## What This Is  PatchGlow is a browser-based modular Eurorack synthesizer emulator. It is an open-source creative coding and art project. It is not a production DAW or a feature-complete synth tool. It is a **beautiful, immersive visual instrument for learning synthesis by doing.**  The full architectural specification lives in `SPEC.md`. Read it before starting any work. It contains the module interface contract, store structure, all module specifications, the visual design system, and the phased build plan.  ## Artistic Vision  This is a cyberpunk / retro-futurist art piece that happens to teach synthesis. The aesthetic is **1982 sci-fi starship console** — think Blade Runner control panels, Alien ship interfaces, TRON circuits, analog oscilloscope phosphor glow. Dark, atmospheric, with neon accents that feel like they are physically emitting light.  The user should feel like they've sat down at a glowing instrument in a dark studio. Every surface is dark matte with subtle texture. Labels glow in cyan, amber, magenta, and purple. Patch cables pulse with signal. Module displays flicker like tiny CRT screens. The whole rack hums with visible energy.  **This is not a clean, flat, modern web app. This is not Material Design. This is not a dashboard.** It should feel handcrafted, analog, warm, and slightly dangerous — like touching a machine that's alive with electricity. Every pixel should serve the atmosphere.  ### Design Principles (ranked by priority)  1. **Visualization is the product.** Every module must have a real-time visual display that reveals what it does to signal. If a user turns a knob and nothing visibly changes on the module's display, the visualization is incomplete. The user learns synthesis by watching — they see the waveform change shape, see the filter curve move, see the envelope trace its path, see the LFO undulate. This is non-negotiable. 1. **Sound is immediate.** Connect two modules with a cable, hear the result. No menus, no setup dialogs, no configuration screens. Drag, patch, listen, learn. 1. **The aesthetic earns attention.** Every element — knobs, cables, panels, text, glows — must feel like it belongs on a retrofuturist spacecraft console. Craft matters. The CRT scanlines, the phosphor glow trails on waveforms, the neon port rings, the cable droop — these details are what make someone stop and want to play with it. Never sacrifice atmosphere for convenience. 1. **Each module teaches one concept.** Oscillator teaches "sound is a waveform." Filter teaches "filtering removes frequencies." Envelope teaches "sound has shape over time." LFO teaches "modulation is one thing controlling another." The module set is curated for pedagogical clarity, not feature count. 1. **Patch cables tell the story.** Cables are color-coded by signal type (amber = audio, cyan = CV, magenta = gate) and animated with flowing particles showing signal direction. The cables are not just wires — they are the visual narrative of how the synth works. A user should be able to look at the cables alone and understand the signal flow.  ## Tech Stack  - **React 18+ with TypeScript** — component model maps to modules - **Vite** — build tool - **Tone.js** — Web Audio framework (the audio engine) - **Zustand** — state management - **@dnd-kit** — drag and drop (library to rack) - **SVG** — patch cable rendering - **Canvas 2D** — per-module visualization displays - **CSS Modules + CSS custom properties** — styling with global theme  ## Key Architecture Decisions  - The **Zustand store** is the single source of truth for rack state (modules, connections, parameters). The audio engine subscribes to it. The UI renders from it. - The **AudioEngineManager** is a singleton that keeps the Tone.js audio graph in sync with the store. - Every module implements the **ModuleRegistration** interface: a definition (ports, parameters, metadata), an audio engine factory, and a visualization component. This is the contribution contract. See `SPEC.md` §3.2 for the full TypeScript interface. - Module visualizations run on **Canvas 2D** at 60fps via a shared `requestAnimationFrame` loop. Each module gets its own canvas element. - Patch cables render as **SVG Bézier curves** with CSS-animated dash patterns for signal flow.  ## Visual Design Quick Reference  **Colors:**  - Background: near-black with blue tint (`#0a0a0f`) - Panels: dark charcoal (`#1a1a24`) - Audio signal: warm amber (`#ff6b35`) - CV signal: cyan (`#00e5ff`) - Gate signal: magenta (`#ff2ecb`) - Each module type has a unique accent color (see `SPEC.md` §4.2)  **Typography:**  - Module labels: Orbitron (or similar geometric sci-fi font), uppercase, letter-spaced, with text-shadow glow - Values and UI text: JetBrains Mono (or similar monospace), clean and technical  **Key visual effects:**  - CRT scanline overlay on module displays (CSS repeating-linear-gradient) - Phosphor glow trails on waveform traces (canvas globalAlpha fade) - Neon glow on port rings, active knobs, and module titles (CSS box-shadow) - Cable droop via cubic Bézier with gravity sag - Subtle vignette on the overall viewport  ## Working On This Project  ### Before starting any task:  1. Read `SPEC.md` if you haven't already 1. Check which build phase the task belongs to 1. Follow the file structure defined in `SPEC.md` §3.5  ### When building a new module:  1. Create a folder in `src/modules/<module-name>/` 1. Implement `definition.ts` (ModuleDefinition), `engine.ts` (ModuleAudioEngine), `Visualization.tsx`, and `index.ts` (ModuleRegistration export) 1. Register the module in `src/engine/moduleRegistry.ts` 1. The visualization must clearly show what the module does — refer to the module's spec in `SPEC.md` §5  ### When making visual/styling decisions:  - Darker is almost always better. When in doubt, go darker. - Glow effects should be subtle and atmospheric, not blinding. Think "phosphor afterglow" not "LED spotlight." - Every interactive element needs visual feedback: knobs glow when hovered/turned, ports pulse when a cable drag is in progress, cables brighten when selected. - Monospace font for all numeric values. Display font for all labels. - No rounded-corner cards, no drop shadows, no white backgrounds, no flat design conventions. This is a hardware instrument, not a web app.  ### Code quality:  - TypeScript strict mode. No `any` types. - Each module must be fully self-contained in its folder. - Tone.js nodes must be properly disposed when modules are removed. - Canvas rendering must not block the main thread — use `requestAnimationFrame`, not `setInterval`. - Test audio in Chrome first (best Web Audio support), then Firefox.  ## Project License  MIT — compatible with all dependencies (Tone.js, Mutable Instruments reference code).  ## Build & Run  ```bash npm install npm run dev     # Vite dev server with HMR npm run build   # Production build to /dist ```  The app is a pure client-side SPA. No backend. Production deployment is static file hosting (Cloudflare Pages, GitHub Pages, or any static server).
